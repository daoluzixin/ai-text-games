# 自反馈工作流

## 适用场景

当任务涉及"代码开发 → 部署 → 验证"这一整套 runtime 验证时阅读。包括但不限于：新增 API 接口、修改业务逻辑、修复 bug、前端功能变更。

## 核心目标

不停留在"代码看起来对"，必须走到"线上环境已验证、有日志/响应证据证明代码确实生效"。

## 标准闭环流程

```
代码修改 → lint 检查 → rsync 推送 → 重启服务 → API/页面验证 → 查日志
    ↑                                                              |
    └──────────── 不通过则修代码，重新走闭环 ←──────────────────────┘
```

### Step 1: Lint 检查

```bash
bash scripts/harness-lint.sh
```

0 FAIL 才允许继续。如果有 FAIL，先修复再往下走。

### Step 2: 推送代码到服务器

```bash
rsync -avz --exclude='.git' --exclude='__pycache__' --exclude='.catpaw' \
  --exclude='.venv' --exclude='sessions.db' --exclude='games/*.docx' \
  -e "ssh -o StrictHostKeyChecking=no" \
  /Users/feng/Downloads/ai-text-games-main/ \
  root@39.106.187.120:/root/ai-text-games/
```

### Step 3: 重启服务

```bash
ssh root@39.106.187.120 'systemctl restart ai-text-games'
```

### Step 4: 验证服务启动

```bash
ssh root@39.106.187.120 'sleep 2 && systemctl is-active ai-text-games && curl -s http://localhost:8080/api/games'
```

期望：服务状态 `active`，API 返回 JSON。

### Step 5: 功能验证

根据本次修改的内容，发送对应的 API 请求或访问页面，确认功能生效。

### Step 6: 查日志确认

```bash
ssh root@39.106.187.120 'journalctl -u ai-text-games --no-pager -n 50'
```

确认无报错，请求正常处理。

## 验证不通过的处理

1. 记录失败现象到 `docs/exec-plans/activeLog/` 对应日志文件
2. 分析原因（代码 bug / 依赖缺失 / 配置问题）
3. 修复代码
4. 从 Step 1 重新走闭环

## 执行日志要求

每次走完闭环，在 activeLog 中记录：

- 本次改了什么
- 验证结果（成功/失败）
- 如果失败：失败原因和修复方案
- commit 信息（如果有提交）
