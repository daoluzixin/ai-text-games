# 执行计划:游戏列表布局优化 + 进入游戏存档选择

## 背景

当前游戏厅首屏布局问题:
1. 顶部独立显示「继续上次的冒险」存档卡片区,占据首屏视觉重心,游戏列表被挤到下方。
2. 游戏卡片偏小(`minmax(160px)`、gap 16px),视觉上密集。
3. 点游戏卡片(`startGame`)会直接进入并自动读档,用户无法选择「重开一局」还是「继续上次」;想重开只能进去后再点「重新开始」。

## 目标

1. 首屏以游戏列表为主,卡片更大、间距更宽松,不密集。
2. 进入某个游戏时:
   - 该游戏**有存档** → 弹窗让用户选「开始新游戏」或「继续上次存档」(显示存档进度预览)。
   - 该游戏**无存档** → 直接进入开始新游戏。
3. 保留删除存档能力(移到选择弹窗内)。

## 方案

仅改前端 `index.html`,**不动后端**(符合「前端零构建」「API 向后兼容」红线)。

### 数据合并
- 并行拉取 `/api/games` 与 `/api/sessions`。
- 按 `game_id` 把存档索引成 map,渲染游戏卡时若命中则在卡片标记「有存档」角标。

### 布局调整
- 移除顶部独立的 `saves-section` 存档区(首屏不再单独陈列存档)。
- 游戏网格:`minmax(160px)` → `minmax(200px)`,gap `16px` → `24px`,卡片 padding 加大,内边距与圆角更舒展。

### 选择弹窗(新增)
- 新增 `choice-modal`:点游戏卡 → 若有存档,展示两个动作卡:
  - 「继续上次存档」:显示上次进度(消息条数 + 最后一句预览 + 时间),点击走原 `loadSessionHistory`。
  - 「开始新游戏」:二次确认会覆盖旧存档 → 调 `/api/reset` 清空后以全新对话进入。
  - 弹窗内提供「删除存档」入口。
- 无存档时跳过弹窗,直接 `enterGame` 进入。

### 流程改造
- 原 `startGame(gameId, name, avatar)` 拆为:
  - `onGameCardClick`:判断有无存档 → 弹窗 or 直接进入。
  - `enterGame(..., mode)`:`mode='continue'` 读档 / `mode='new'` 清档后进入。

## 验证

1. 本地起服务,浏览器查看:列表卡片间距、有存档游戏的角标、点击弹窗交互。
2. 跑 `bash scripts/harness-lint.sh`,0 FAIL。

## 影响面

- 仅 `index.html`。后端、API、数据库均不变。
