# 执行计划：选项卡片点击直接发送

## 背景 / 目标

XML 渲染修复后，AI 回复里的 `<choices>` 已能渲染成 A/B/C 选项卡片。
用户希望**交互更友好：点击选项卡片就直接把该选项作为回复发给 AI**，
无需手动把选项内容打字到输入框。

## 现状

- `static/md-frame.html`：choices 在两处渲染（严格路径 `renderXmlStrict` ~308 行、
  宽松路径 `renderXmlLoose` ~367 行），均输出 `<div class="xml-choice">`，无点击能力。
- `index.html`：
  - 每条 AI 消息的 iframe 都注册了 `onMsg` 监听器（处理 `md-frame-ready` / `md-frame-height`）。
  - 已有 `sendQuick(text)`：填入 `#chat-input` 并调用 `sendMessage()`。
  - `sendMessage()` 内有 `isStreaming` 卫语句，流式进行中点击会被忽略（天然防重复）。

## 方案（最小改动，复用现有链路）

### 1. md-frame.html —— 选项可点击 + 上报
- 渲染 choice 时加 `data-choice-id` 和 `data-choice-text`（文本做属性转义），
  并加 class 标记可点击（CSS 已有 hover 态，光标改 pointer）。
- 用**事件委托**在 `#content` 上监听 click：命中 `.xml-choice` 时，
  读取其 `data-choice-id` / `data-choice-text`，
  `window.parent.postMessage({ type: 'md-choice', id, text }, '*')`。
  - 用委托而非逐个绑定，避免 innerHTML 重渲染后监听丢失。
- 严格路径与宽松路径生成的 choice DOM 结构保持一致，复用同一委托。

### 2. index.html —— 监听并发送
- 在每条 AI iframe 的 `onMsg` 里增加分支：`e.data.type === 'md-choice'`：
  - 组装发送文本：优先 `「id」. text`（如 `A. 端庄行礼…`），text 为空则只发 id。
  - 调用 `sendQuick(payload)`（已含填入+发送+isStreaming 防重）。
- 仅当前消息 iframe 的监听响应（`e.source !== iframe.contentWindow` 已过滤），
  避免历史多条 AI 消息的 iframe 重复触发。

## 不做 / 红线
- 不改后端 `server.py`（纯前端交互增强，API 不变）。
- 不动 XML 渲染主逻辑、status 卡片、前导剥离逻辑。
- 历史存档同样受益（回填的 choices 也可点）。

## 验证
1. lint：`bash scripts/harness-lint.sh` 须 0 FAIL。
2. Node 纯函数验证：choice DOM 含 data-choice-id/text 且文本正确转义。
3. 浏览器实测：点击选项 → 输入框被填入 → 自动发送 → AI 流式响应；
   流式进行中点击不重复发送。
4. 部署后线上 curl 校验 md-frame 含 `md-choice` postMessage 标记。
